# E-Shop Pro 项目记忆（最新更新：2026-02-20）

## 1. 项目基本信息
- 项目名称：E-Shop Pro
- 目标用户：仅中国大陆消费者
- 支持语言：仅简体中文
- 支持货币：仅人民币（CNY）
- 平台类型：B2C 多商家入驻电商平台（初期支持商家入驻，但审核流程简化）
- 当前阶段：阶段1 - 需求分析和规划（刚启动）
- 预计总周期：9-12个月（小团队，只有老板 + 总工程师两人）
- 云平台：优先阿里云（ECS / RDS MySQL / OSS 对象存储 / CDN / SMS 等服务）
- 项目代号/关键词：E-Shop Pro、小团队电商、Spring Boot + Next.js、中国支付

## 2. 已确认的技术栈（第一版，MVP阶段）
- 前端：
  - 框架：Next.js（推荐使用 App Router，支持 SSR/SSG，提升 SEO 和首屏速度）
  - UI 组件库：Ant Design（中文支持完善，组件丰富）
  - 状态管理：Redux（或 Redux Toolkit），必要时结合 Zustand 简化部分场景
  - 移动适配：PWA（Progressive Web App），初期不做独立 App
- 后端：
  - 语言/框架：Java + Spring Boot 3.x（单体架构，MVP阶段不拆微服务，降低复杂度）
  - API 风格：RESTful（JSON 接口）
  - 认证授权：JWT（无状态） + 微信/支付宝一键登录（OAuth2）
- 数据库：
  - 主数据库：MySQL 8.x（阿里云 RDS）
    - 存储：用户、商家、商品基本信息、SKU、库存、订单、支付记录、地址等
    - 商品复杂属性/描述：使用 JSON 字段（MySQL JSON 类型）或商品属性扩展表（初期优先 JSON 字段简化）
  - 缓存：Redis（用于购物车、热点商品、Session 等，阿里云 Redis）
  - 全文搜索：初期不引入 Elasticsearch，后期商品搜索量大时考虑添加
- 支付集成：
  - 支付宝（官方 SDK，当面付/手机网站支付/APP支付）
  - 微信支付（官方 SDK，JSAPI/小程序支付/APP支付）
  - 支付回调：异步通知 + 主动查询对账
- 消息/通知：
  - 短信：阿里云 SMS（注册、登录、订单状态变更）
  - 异步任务：RabbitMQ（订单创建后发通知、库存扣减等）
- 部署 & DevOps：
  - 服务器：阿里云 ECS（初期 2核4G 起步）
  - 域名 & HTTPS：阿里云解析 + 免费/付费 SSL
  - CI/CD：GitHub Actions（简单自动化构建、部署）
  - 监控：Spring Boot Actuator + 阿里云 ARMS（应用实时监控服务）
- 其他工具：
  - 版本控制：Git + GitHub（私有仓库）
  - 测试：JUnit 5（后端） + Jest + React Testing Library（前端）
  - 文档/协作：本记忆文档 + 飞书/语雀（需求、接口文档）

## 3. 当前功能范围（MVP核心，必须实现）
- 用户端（H5网页 + PWA）：
  - 手机号注册/登录 + 微信一键登录
  - 商品首页、分类导航、搜索（关键字 + 简单筛选）
  - 商品详情页（图片、规格、价格、库存、评价展示）
  - 购物车（添加/编辑/删除/选中结算）
  - 地址管理（新增/编辑/默认地址）
  - 提交订单 → 选择支付方式（支付宝/微信） → 支付 → 支付成功页
  - 我的订单列表 + 订单详情（查看物流、申请退款等）
- 商家/管理员端（PC 为主，H5 可访问）：
  - 商家入驻（简单表单 + 人工审核，初期不做复杂资质）
  - 商品管理：上架/编辑/下架、规格/SKU、库存管理、图片上传（用阿里云 OSS）
  - 订单管理：查看订单、发货（输入物流单号）、退款处理
  - 简单数据统计（今日订单数、销售额）
- 后台通用：
  - 管理员登录（超级管理员可管理商家）
  - 基础权限控制（商家只能看自己店铺数据）

暂不包含（MVP明确排除，节省工作量）：
  - 多语言/多币种
  - 直播购物、秒杀、拼团、优惠券、积分
  - 复杂推荐算法（猜你喜欢等）
  - 原生 App（先 PWA 顶替）
  - 微信小程序（可作为后续迭代）

## 5. 重要决策 & 变更历史
- 2026-02-20：决定采用单体 Spring Boot 架构（而非微服务），因为团队只有两人，优先降低部署/维护复杂度
- 2026-02-20：支付方式最终锁定为支付宝 + 微信支付（放弃 Stripe/PayPal，不适合中国国情）
- 2026-02-20：数据库选型 MySQL 而非 PostgreSQL（中国电商生态更成熟、阿里云优化更好、学习/招聘成本低）
- 2026-02-20：商品复杂属性暂时全部用 MySQL JSON 字段存储（不引入 MongoDB，简化初期架构）
- 2026-02-20：团队组成确认只有老板 + 总工程师两人，上线后再考虑招聘

## 6. 待澄清 / 待决策事项（老板重点关注）
- 商品主要品类是什么？（服装、数码、食品、日用等？影响图片/规格设计）
- 是否允许游客（未登录）浏览商品 + 加购物车？（只在结算时强制登录？）
- 商家入驻需要哪些信息？（姓名、身份证、店铺名、联系方式？是否需要上传营业执照？）
- 商品图片/用户上传头像存储方案：确认用阿里云 OSS（推荐）
- 是否需要支持退货地址管理？（商家设置统一退货地址？）
- 首页/分类页是否需要 banner 轮播图？由谁维护（管理员后台？）
- 物流对接：初期是否只记录物流单号（手动），还是要接入快递鸟/顺丰 API？
